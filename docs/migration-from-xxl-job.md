# XXL-JOB 迁移到 XXL-JOB Boost 指南

## 目标

这份文档说明三类迁移路径：

- 官方 `XXL-JOB 3.x` 迁移到 `XXL-JOB Boost`
- 官方 `XXL-JOB 2.4.x / 2.5.x` 先迁移到 `3.x`，再迁移到 `Boost`
- 官方 `XXL-JOB 2.3.x 及更早版本` 先迁移到 `2.4.x / 2.5.x`，再迁移到 `3.x`，最后迁移到 `Boost`

文档默认目标是：

- 保留现有任务、执行器和数据库
- 尽量不重建库
- 尽量以分阶段、可回滚的方式上线

## 先说结论

### 推荐迁移路径

| 当前版本 | 推荐路径 |
| --- | --- |
| 官方 `3.x` | `3.x -> XXL-JOB Boost` |
| 官方 `2.4.x / 2.5.x` | `2.4.x / 2.5.x -> 官方 3.x -> XXL-JOB Boost` |
| 官方 `2.3.x 及更早` | `2.3.x 及更早 -> 2.4.2 或 2.5.0 -> 官方 3.x -> XXL-JOB Boost` |

### 为什么这么拆

- 官方文档明确说明：`v3.0.0` 开始，数据模型和通讯协议对 `v2.4.*` 及后续版本向前兼容，可以无缝升级。
- 官方文档没有承诺 `2.3.x` 及更早版本可以直接无缝升级到 `3.x`。
- 官方 `v2.5.0` 是基于 `JDK 8` 的最后一个大版本，适合作为老 2.x 系统过渡站。
- `XXL-JOB Boost` 当前代码基线是 `1.0.0`，并且仓库要求 `JDK 17+`。

## 版本边界

### 官方 2.4.x / 2.5.x

- `v2.4.0` 发布于 `2023-03-23`
- `v2.4.1` 发布于 `2024-04-17`
- `v2.4.2` 发布于 `2024-11-16`
- `v2.5.0` 发布于 `2025-01-11`

其中 `v2.5.0` 官方备注有两点很关键：

- `v2.4.*` 代码和系统可无缝升级到 `v2.5.x`
- `v2.5.x` 是基于 `JDK 8` 的最后一个大版本

### 官方 3.x

- `v3.0.0` 发布于 `2025-02-07`
- 官方明确要求：`3.x` 开始需要 `JDK 17`
- 官方明确说明：`v2.4.*` 及后续版本可无缝升级到 `v3.0.0`

### XXL-JOB Boost

当前仓库基线：

- 版本：`1.0.0`
- Java：`JDK 17+`
- 后端：`Spring Boot 4`
- 新旧控制台并存：
  - 旧控制台：`/xxl-job-admin/`
  - 新控制台：`/xxl-job-admin/admin-next/`

Boost 不是脱离官方 3.x 重新设计的一套系统，而是在 `XXL-JOB 3.x` 演进线上的增强发行版。

## 迁移前检查

正式迁移前，建议先做这几件事：

1. 记录当前版本号、JDK 版本、部署方式。
2. 备份数据库。
3. 备份调度中心配置文件。
4. 备份执行器配置文件。
5. 导出任务、执行器、用户和权限清单。
6. 记录 `accessToken`、执行器 `appname`、端口、注册地址等关键参数。
7. 先在预发环境完成一次全流程演练。

建议至少准备下面这些回滚材料：

- 旧版 admin 包
- 旧版 executor 包
- 旧版配置
- 数据库快照或可回放备份

## 路径一：官方 3.x 迁移到 XXL-JOB Boost

这是最简单、风险最低的一条路径。

### 适用范围

- 你当前已经在官方 `3.0.x`、`3.1.x`、`3.2.x`、`3.3.x` 或同代版本
- 当前运行环境已经满足 `JDK 17+`

### 迁移步骤

#### 1. 确认运行时基线

确认以下条件已经满足：

- 调度中心运行在 `JDK 17+`
- 执行器如果跟随 `3.x` 官方版本，也建议统一到 `JDK 17+`
- 当前数据库字符集、索引和表结构已稳定

#### 2. 对比数据库结构

先拿你的现网库和本仓库初始化 SQL 做 diff：

- 基线文件：`doc/db/tables_xxl_job.sql`

如果你不是从 Boost 初始化的数据库，而是从官方 XXL-JOB 库升级过来的，至少按文件名顺序执行并确认下面增量已经补齐：

- `doc/db/migrations/2026-06-13-add-alarm-rule-table.sql`
- `doc/db/migrations/2026-06-13-add-operator-user-id-to-audit-log.sql`
- `doc/db/migrations/2026-07-04-upgrade-to-xxl-job-boost-1.0.0.sql`

说明：

- 这些脚本按幂等方式编写，可重复执行
- 如果你的库和 `doc/db/tables_xxl_job.sql` 相比还有别的差异，需要先在预发库确认原因，再决定是否额外补齐

#### 3. 替换调度中心

部署 `xxl-job-boost` 的 `xxl-job-admin`：

- 保留原有 context path：`/xxl-job-admin`
- 保留原有数据库连接
- 保留原有登录入口

这样可以保证：

- 老控制台继续可用
- 新控制台可以直接作为新入口试跑

#### 4. 先升级 admin，再观察 executors

如果你当前 executors 也是 `3.x`，通常可以先不动执行器，先替换调度中心观察。

推荐验证项：

- 任务列表是否完整
- 手动触发是否正常
- 回调与日志是否正常
- 执行器注册是否正常
- 老控制台是否仍然可访问
- 新控制台 `admin-next` 是否可访问

#### 5. 分批切换使用入口

推荐顺序：

1. 先让管理员试用新控制台
2. 再逐步把日常操作迁到新控制台
3. 旧控制台保留一段时间作为回退入口

## 路径二：官方 2.4.x / 2.5.x 迁移到 3.x，再迁移到 Boost

这是最推荐的 2.x 迁移方案。

### 为什么不建议 2.4.x 直接跳到 Boost

理论上，Boost 本身就在 `3.x` 演进线上，很多场景可以直接切。

但默认仍建议拆成两步：

1. `2.4.x / 2.5.x -> 官方 3.x`
2. `官方 3.x -> Boost`

原因是：

- 这样能先验证“官方升级链路”本身没问题
- 出问题时更容易判断是官方 2.x -> 3.x 的兼容问题，还是 Boost 自身增强带来的问题
- 更适合生产环境分阶段控制风险

### 推荐中转版本

如果你当前是 `2.4.x`，建议先统一到：

- `2.4.2`，或者
- `2.5.0`

更推荐 `2.5.0`，因为它是官方明确标注的 `JDK 8` 最后一个大版本，且已经包含：

- 多项漏洞修复
- 日志索引优化
- 更稳定的 2.x 末期基线

### 迁移步骤

#### 阶段 A：先升级到 2.4.2 或 2.5.0

如果当前已经是 `2.4.2` 或 `2.5.0`，这一步可以跳过。

如果当前是更早的 `2.4.0 / 2.4.1`，建议先把：

- admin
- executor
- 依赖包

统一到 `2.4.2` 或 `2.5.0`，并完成一轮验证。

#### 阶段 B：准备 JDK 17

从 `2.x` 到 `3.x` 的核心变化之一不是数据库，而是运行时。

你需要先准备：

- 调度中心 `JDK 17`
- 执行器 `JDK 17`
- 镜像、脚本、容器基线同步切换到 `JDK 17`

这一阶段最容易出问题的不是 XXL-JOB 本身，而是你自己的业务执行器依赖。

重点检查：

- 老 Spring Boot 版本是否兼容 `JDK 17`
- 自定义执行器里是否用到了过时反射、JAXB、旧版驱动
- 部分业务任务是否依赖本机命令或路径

#### 阶段 C：升级到官方 3.x

建议先升级到一个稳定的官方 `3.x` 版本，完成基本回归：

- 执行器注册
- 手动触发
- Cron 触发
- 回调入库
- Rolling Log
- 登录和权限

这一步如果通过，说明：

- 你的库已经适配 `3.x`
- 你的执行器也已经完成 `JDK 17` 切换

#### 阶段 D：从官方 3.x 切到 Boost

完成上一步后，再按“路径一”切到 `XXL-JOB Boost`。

## 路径三：官方 2.3.x 及更早版本迁移

这类版本不要直接尝试跳 `3.x`。

### 原因

官方明确承诺的无缝升级起点是：

- `v2.4.*`

不是：

- `v2.3.x`
- 更老的 `2.2.x / 2.1.x / 1.x`

### 推荐路径

`2.3.x 及更早 -> 2.4.2 或 2.5.0 -> 官方 3.x -> Boost`

### 详细步骤

#### 阶段 A：先升级到 2.4.2 或 2.5.0

这一步的目标不是直接上新功能，而是先落到官方承诺可继续无缝升级的起点版本。

更推荐 `2.5.0`，原因：

- 它是 `JDK 8` 最后大版本
- 官方明确建议后续大版本 `3.0` 将基于 `JDK 17`
- 它已经吸收了较新的安全修复和索引优化

#### 阶段 B：完成 2.x 末期自检

重点检查：

- `accessToken` 是否已统一配置
- 日志表是否有历史大表性能问题
- 旧执行器是否存在非常老的 starter 或通信依赖
- 自定义扩展是否直接改过 XXL-JOB 源码

如果你们曾经做过深度定制，这一步要额外谨慎。

#### 阶段 C：切换到 JDK 17 并升级官方 3.x

这一步和“路径二”的阶段 B、C 一样处理。

#### 阶段 D：再迁移到 Boost

等官方 `3.x` 已经稳定后，再切到 `XXL-JOB Boost`。

## 数据库迁移建议

### 原则

- 不要直接假设“只替换 jar 就够了”
- 每次升级都要比对当前库结构和目标版本初始化 SQL
- 生产库升级优先走增量 SQL，不直接重建

### Boost 当前建议检查项

至少检查以下对象是否存在并符合当前仓库定义：

- `xxl_job_alarm_channel`
- `xxl_job_alarm_rule`
- `xxl_job_alarm_record`
- `xxl_job_audit_log`
- `xxl_job_info` 中的告警与标签相关字段
- `xxl_job_audit_log.operator_user_id`

当前仓库明确提供的增量脚本：

- `doc/db/migrations/2026-06-13-add-alarm-rule-table.sql`
- `doc/db/migrations/2026-06-13-add-operator-user-id-to-audit-log.sql`
- `doc/db/migrations/2026-07-04-upgrade-to-xxl-job-boost-1.0.0.sql`

如果你是从官方库直接迁过来，建议把你的现网库与 `doc/db/tables_xxl_job.sql` 做一次结构对比，并在预发环境完整演练上述迁移链。

## 迁移过程中的验证清单

每一个阶段完成后，至少验证下面这些项目：

### 调度中心

- 可以正常登录
- Dashboard 指标正常
- 任务列表、执行器列表、日志列表可加载
- 新旧控制台都可访问

### 执行器

- 自动注册正常
- 手动触发正常
- Cron 正常触发
- 回调正常
- 日志可查看

### 权限与安全

- 管理员和普通用户权限正常
- `accessToken` 配置没有丢
- Webhook、邮箱或其他告警通道配置可用

### 数据

- 历史任务仍在
- 历史日志可查
- 用户、权限、执行器配置未丢失

## 推荐上线方式

### 方案一：先预发完整演练，再生产切换

适合大多数团队。

做法：

1. 复制生产库到预发
2. 完整跑一遍升级
3. 预发联调通过后，再安排生产窗口

### 方案二：生产双入口观察期

适合已经迁到官方 `3.x`，再切 Boost 的场景。

做法：

1. 切换到 Boost 后先保留旧控制台访问入口
2. 让管理员优先使用 `admin-next`
3. 观察一段时间后再决定是否进一步收口

## 不建议的做法

- 从 `2.3.x` 直接跳到 Boost
- 不切 `JDK 17` 就尝试上线 `3.x` 或 Boost
- 不做数据库备份直接替换生产包
- 不做预发验证就直接升级调度中心和执行器
- 一次性同时改 XXL-JOB 版本、JDK 版本、数据库结构和业务执行器逻辑

## 建议的最小风险实施顺序

如果你要一个最稳的实际执行顺序，建议照这个来：

1. 盘点当前 XXL-JOB 版本、JDK、数据库和执行器依赖
2. 备份数据库和配置
3. 如果低于 `2.4.x`，先升到 `2.4.2` 或 `2.5.0`
4. 把运行时和执行器准备到 `JDK 17`
5. 升级到官方 `3.x`
6. 验证调度、回调、日志、权限
7. 对照 `doc/db/tables_xxl_job.sql` 补齐 Boost 所需结构
8. 部署 `XXL-JOB Boost`
9. 先用旧控制台兜底，新控制台逐步切流
