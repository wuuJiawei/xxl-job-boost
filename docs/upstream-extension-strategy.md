# XXL-JOB Boost 源码增强策略

## 目标

在 `xxl-job` 现有源码基础上直接做增强，保持改动集中、可读、可维护，同时接受未来同步 upstream 时出现 merge 冲突这一现实成本。

核心原则：

1. 直接在现有模块上演进，不额外为了“零侵入”强拆模块
2. 新能力优先以新增代码、新页面、新接口的方式接入
3. 允许并接受对原有模块、控制器、服务、Mapper 的必要修改
4. AOP / 拦截器只解决横切问题，不承载主业务

## 推荐路线

采用“源码内渐进增强”的方式，而不是为了后续升级成本，把当前系统过早拆成插件化扩展层。

当前建议结构：

- `xxl-job-core`
  - 核心调度能力
  - 允许按增强目标直接修改
- `xxl-job-admin`
  - 管理后台与后端接口
  - `admin-next` 的 controller / API / 静态资源直接放在这里
- `xxl-job-admin-ui`
  - 前端源码工作区
  - 构建产物输出到 `xxl-job-admin`

## 代码边界

### 可以做的

- 新增 controller / config / service
- 新增静态资源和 SPA
- 新增扩展文档、脚本、前端工程
- 修改现有 controller / service / mapper / model
- 修改现有调度、注册、推送相关实现

### 倾向避免的

- 没有明确收益的“纯结构性重构”
- 为了规避未来 merge 冲突而引入当前复杂度
- 把简单需求过度设计成插件体系
- 把主业务流程塞进 AOP / 拦截器里

### 修改原则

如果必须改原模块，遵守下面几点：

1. 优先新增而不是覆盖旧代码
2. 能在局部闭环解决，就不要扩散修改面
3. 改动原因要清晰，别留下“为什么这里突然变了”的黑盒
4. 新旧控制台并存期间，避免破坏旧页面主链路

## AOP 使用边界

适合 AOP 的场景：

- 审计日志
- 埋点
- 限流
- 权限补充校验
- 统一异常包装

不适合 AOP 的场景：

- 新页面
- 新接口协议
- 新查询模型
- 表单编排
- 复杂流程替换

## 当前落地方向

已采用的方向：

- `admin-next` 直接放在 `xxl-job-admin`
- `xxl-job-admin-ui` 构建产物输出到 `xxl-job-admin/src/main/resources/static/admin-next`
- 后续自动注册、推送增强等能力，允许直接进入原模块源码修改

## 后续执行规则

后续需求默认按下面顺序评估：

1. 能否通过新增页面 / 新增接口 / 新增类解决
2. 如果不能，能否在现有模块局部修改解决
3. 如果还不能，再评估是否需要更大范围的结构调整

如果改动原有核心逻辑，需要单独说明：

- 为什么需要改
- 最小改动面是什么
- 未来同步 upstream 时大概会冲击哪里
